# 実投稿による検証メモ

## 何を検証できるか

`xalgo` が計算するのは公開カウントから作った群集平均の代理スコアであり、
閲覧者別のPhoenix予測ではない。この実験で確認できるのは、代理スコアと
公開された到達・第三者ランキングの関係だけである。Xの内部スコアとの一致を
直接証明する実験ではない。

## データ

- 標本: [XBeast “Today's Viral Tweets”](https://xbeast.io/viral-tweets) の
  2026-07-20 UTCスナップショット9件と、比較用2件
- ライブ取得: 2026-07-21 12:27 UTC
- 取得方法: `scripts/validate_popular.py` からFxTwitter等の公開バックエンドを使用
- スコア: `repo_demo`（ラベル上はfavorite 1.0 / reply 0.5 / retweet 0.3 /
  dwell 0.2）。後の契約監査で`run_pipeline.py`のindexと`runners.py`の出力head順に
  不整合が見つかったため、これは正解重みではなく感度分析用の仮定として扱う
- 注意: XBeastのVirality Scoreは第三者の独自指標で、Phoenixスコアではない

公開カウントは時間とともに変化するため、以下は一時点の結果である。手元では
`python scripts/validate_popular.py --json` で取得時刻を含むJSONを再生成できる。

## 解釈

この小さな選択標本では、観測エンゲージメント率が高い投稿ほどviewsが多い、
という関係は見られなかった。考えられる理由は次のとおり。

1. **標本選択**: すでにバイラルになった投稿を第三者ランキングから選んでいる。
   到達が大きいという条件で選別した後の相関は、一般投稿の相関を表さない。
2. **希釈**: 投稿がコア読者の外まで配信されると、累積のlikes/views等が下がり得る。
3. **時点の不一致**: 公開カウントは取得時点までの累積値で、配信判断時点の
   Phoenix予測とは異なる。
4. **不可視シグナル**: dwell、VQV、共有、ネガティブ反応など、公開カウントから
   作れない信号が多い。
5. **パーソナライズ**: Phoenixの確率は閲覧者履歴に依存するが、公開カウントは
   全閲覧者を混ぜた事後集計である。

小規模アカウントの投稿がフォロワー数を大きく超えて到達している点は、
out-of-network候補を扱う公開設計とは矛盾しない。ただし、それだけでランキング式や
重みが実運用と一致しているとは言えない。今回の負の相関は「代理値の限界」を示す
結果であり、実アルゴリズムの正しさ・誤りのどちらも確定しない。

## 次に行う検証

1. 投稿後15分、30分、1時間、2時間の公開カウントを同じURLで定点観測する。
2. 投稿時刻、言語、メディア種別、views帯を揃え、選択バイアスを減らす。
3. 自分のFor You画面で同時に表示された候補集合と順位を、同意を得た上で記録する。
4. 順位との比較は単純相関だけでなく、pairwise accuracyとNDCGでも測る。
5. 十分な標本数と複数アカウントで再現性を確認する。
